# 0007. 産出へ転回 —「単語が語尾に化けた」ことが決め手

- 日付: 2026-07-20
- ステータス: 有効
- 関連: [0006](./0006-prediction-came-online.md), [0005](./0005-katakana-is-accurate-transcription.md), [0003](./0003-dark-l-is-the-real-blocker.md)

## 文脈

レッスン 06 の宿題（枠を1つ見つける）の報告:

- **枠の発見に成功**: `What I recommend you do is`
- 残った誤り: `you go to the top here` → 「goes the top here」

[0006](./0006-prediction-came-online.md) の判断基準「枠を自力で見つけられた → 産出へ移行」を満たしたので、**宣言どおり産出へ転回した**。[0003](./0003-dark-l-is-the-real-blocker.md) から4回持ち越していた領域。

## 診断

| 実際 | 音 | 処理 |
|---|---|---|
| `is you` | [ɪʒu] | 丸ごと消失（/z/+/j/ の融合） |
| `go to` | [ɡoʊ**ɾ**ə] | **`goes` に統合** |

`go to` の /t/ は母音間 + 後続非強勢で、**レッスン 05 の flap 条件そのもの**。ユーザーは [ɾə] を検出したが、**語として復元せず活用語尾 -s として吸収した**。`goes` が実在語形なので辻褄が合ってしまった。

## 非自明な洞察

**これが今回の最重要点。**

1. **1語が1形態素に化けた。** レッスン 05 で渡したのは「ラ行 → /t/」という**音素レベル**の逆変換だったが、今回失敗したのは**着地先の階層の選択**（語彙 vs 形態論）。音素は正しく戻せている。**規則を1本渡すと、その規則の適用先を誤るという新しい誤りが出る** —— これは [0006](./0006-prediction-came-online.md) の「規則を渡した次の回に反例が来る」の3回目で、もはやパターンとして扱うべき。単純化して渡す方針自体は有効だが、**次回必ず補正する前提で設計する**。
2. **これは聞く練習では直りにくい。** ユーザーは `to` を [tuː] としてしか発音したことがない。だから [ɾə] を `to` の候補として**検索しない**。検索範囲の問題であって、聴覚の問題ではない。**産出が検索範囲を広げる**という因果があるので、ここで産出に移るのは順序として正しい。shadowing は理解可能性・流暢性に加え**リズム・イントネーション・弱形の産出**を伸ばす（[systematic review, 44件](https://www.tandfonline.com/doi/full/10.1080/29984475.2025.2546827)）。まさに弱形の話。
3. **枠は `is` の1語先まで続く。** [0006](./0006-prediction-came-online.md) で「情報は `is` の後ろ」と教えたが不正確だった。実例2件とも軸の後ろに**主語を繰り返す代名詞**が入る（`is it installs` / `is you go`）。助走路は **`is` + 代名詞**まで。
   - **ただし典拠を確認できていない。** 動詞分布（`do` 66%）のような統計は見つからなかった。レッスン内で「規則ではなく傾向」「ユーザーの実例2件からの観察」と明示した。**裏が取れていないものは取れていないと書く**——教材の信頼性はここで決まる。
4. **産出には自動フィードバックループを作れる。** SpeechRecognition で発話を文字化し、目標文と語単位で照合できる。さらに `speechstart`/`speechend` の差分で**所要時間**が取れるので、**圧縮できているかを直接評価できる**。産出の核心は語を並べることではなく予算配分なので、時間を評価対象にしたのは本質的。ASR を使った shadowing 練習は先行研究もある（[Journal of Computers in Education](https://link.springer.com/article/10.1007/s40692-025-00374-x)）。

## 実装したこと

- **`assets/speaking.js` を新規作成**（この workspace 初の産出部品）
  - お手本再生（`speechSynthesis`）→ 発話 → **語一致率 + 所要時間**で判定
  - `maxAlternatives = 3` を取り、目標に最も近い候補を採用（認識器の第一候補は言語モデルに整形される）
  - 完全一致ではなく**含有率**で判定（認識器は冠詞等を落とす）
  - **非対応環境では自己採点チェックリストに退避**して練習は続行できる
  - `dictation.js` がグローバルに `const normalize` を定義しているため、**IIFE で包んで衝突を回避**した
- `assets/style.css` に `.speaking` 系と `.beat` / `.flat`（強勢の可視化）を追加。印刷時は操作ボタンを隠す
- `reference/speaking-rhythm.html` を新規作成（予算配分、縮約の常用形9種、shadowing 手順）
- 宿題を**聞く → 話す**に変更。報告してもらうのは「聞き取れなかった箇所」ではなく**「口が回らなかった箇所」**

## 未検証の前提

- **`speaking.js` は未実機検証。** ユーザー環境（Chrome / macOS）で動く見込みだが、マイク許可・ネットワーク依存の認識・時間計測の精度、いずれも確認できていない。**レッスン内でユーザーに動作報告を明示的に依頼した。** 動かなければ設計を変える
- 時間予算（1.4〜3.2秒）は**私の見積もりで、実測に基づいていない**。厳しすぎる可能性がある。初回の結果で較正する
- 反復代名詞の一般性（上記 3 の典拠問題）

## 次の判断ポイント

- 判定が動き、時間予算も達成 → 枠の産出は済み。**自由産出**（自分の設計判断を英語で説明する）へ
- 語は言えるが時間が足りない → 圧縮の問題。縮約の常用形を口で作る練習を増やす
- 判定部品が動かない → 録音＋自己採点、あるいは対人（コミュニティ）に切り替える
- 「口が回らない箇所」が特定の音（例: /r/, θ, 語末子音連結）に偏る → 調音の個別指導へ
